# Feature List

Tracks progress against the constitution's Development Workflow gate:
every feature below needs a complete `spec.md`, `plan.md`, and `tasks.md`
before implementation begins on **any** of them (v0.3.0-draft). This is a
living tracking doc, not itself a spec — update it as wireframes land and
as `/speckit-specify`/`/speckit-plan`/`/speckit-tasks` are run per feature.

**Status key**: 🎨 wireframe pending · ✅ wireframed, not yet spec'd ·
📝 spec/plan/tasks in progress · 🟢 ready (spec+plan+tasks all complete) ·

New `conversations`/`messages` tables — `conversations.lastReadAt` (a per-member JSON read-cursor, added beyond data-model.md's original two-column sketch) is the actual mechanism behind accurate unread counts. `accept-request.ts` is this project's first `db.transaction()`, guarded by a `.returning()`-checked update so a concurrent double-accept can't double-decrement `seatsOpen` or create two conversations (verified with a real concurrent-call test, unlike the toggle-race antipattern Forum Thread's own lesson warned against — accept is a one-way transition, not a toggle, so concurrent calls here are a valid test). `search-contacts.ts` excludes blocked/blocking users in *either* direction (Blocked Users' own candidate search only excludes one direction), and `start-conversation.ts` re-checks the same relationship server-side. Found and fixed three real bugs: (1) a native `<dialog>` styled with a bare Tailwind `flex` utility never actually hides when closed, because Tailwind v4 emits utilities inside `@layer`, and per the CSS cascade-layers spec, layered author styles always beat the UA stylesheet's `dialog:not([open]) { display: none }` regardless of specificity — fixed via `hidden open:flex`, a latent bug likely also present in Blocked Users'/Forum index's own dialogs, left unfixed there (out of this feature's scope) but worth a follow-up; (2) this Next.js version's client `router.refresh()` called immediately after `router.push()` can lose a race and revert navigation to the pre-push URL — confirmed via server logs showing the target route's fetch succeeding while the browser still showed the old URL; fixed by moving cache invalidation server-side into the Server Actions themselves via `revalidatePath(path, "layout")` (this Next.js version's documented pattern, `next/cache`'s `refresh()`/`revalidatePath` are Server-Action-only now), with client code doing a bare `router.push()` and nothing else; (3) a Playwright `getByText()` false-positive matched a composer `<textarea>`'s still-pending typed value before its async send resolved — fixed by waiting for the textarea to clear first.
